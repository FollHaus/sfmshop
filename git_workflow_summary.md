# Git Workflow для проекта SFMShop

## Использованные команды Git

- `git init` — инициализация локального репозитория.
- `git clone` — клонирование удалённого репозитория (подразумевается, так как есть подключение к `origin`).
- `git add <file>` — добавление изменений в staging area (например, `git add src/models/product.py`).
- `git commit -m "сообщение"` — создание коммита с описанием изменений.
- `git branch` — просмотр списка локальных веток.
- `git branch -a` — просмотр всех веток (локальные и удалённые).
- `git checkout <branch>` — переключение между ветками.
- `git checkout -b <new-branch>` — создание новой ветки и переход на неё.
- `git merge <branch>` — слияние указанной ветки в текущую.
- `git pull` — получение изменений из удалённого репозитория (`git fetch + git merge`).
- `git push -u origin <branch>` — отправка локальной ветки в удалённый репозиторий с установкой отслеживания.
- `git log --oneline --graph --all` — визуализация истории коммитов.
- `git log --stat` — просмотр истории коммитов с указанием изменённых файлов и количества строк.

## Созданные ветки

- **main** — основная стабильная ветка проекта. Содержит рабочую версию кода, доступную для деплоя.
- **feature/add-discount-system** — ветка для реализации системы скидок в классе `Product`. Реализовано ранее, слита в
  `main` через Pull Request.
- **feature/add-inventory-management** — ветка для добавления методов управления складом (например, учёт остатков).
  Успешно слита в `main`.
- **feature/add-shipping** — ветка для реализации расчёта стоимости доставки (`calculate_shipping`). Изменения были
  объединены с `main`, где уже был добавлен `get_category`.
- **feature/add-logging** — ветка для внедрения системы логирования. Не слита, находится в разработке.
- **feature/add-caching** — ветка для добавления кэширования (например, через `functools.lru_cache`). Находится в
  разработке.

## Разрешенные конфликты

- **Конфликт в `src/models/product.py` при слиянии `feature/add-shipping` и `main`**:
    - Причина: обе ветки модифицировали один и тот же файл — `src/models/product.py`.
        - Ветка `feature/add-shipping` добавила метод `calculate_shipping(self)`.
        - Ветка `main` (через другую feature-ветку) добавила метод `get_category(self)`.
    - Решение: ручное объединение методов в один файл без потерь. Оба метода были сохранены.
    - Команда: `git merge main` в ветке `feature/add-shipping`, затем редактирование файла, `git add`, `git commit`.
    - Итог: успешный мёрдж-коммит `97166c7`.

## Стратегия работы с ветками

Проект использует **Feature Branch Workflow**:

1. Основная ветка — `main`, защищена от прямых коммитов.
2. Каждая новая функция разрабатывается в отдельной ветке (`feature/...`).
3. После завершения разработки создаётся Pull Request (PR) в GitHub.
4. Код проходит ревью, при необходимости — доработку.
5. После одобрения PR вливается в `main`.
6. После мержа, локальная ветка может быть удалена, а `main` обновляется через `git pull`.

## Workflow для проекта SFMShop

1. Разработчик создаёт ветку: `git checkout -b feature/add-logging`
2. Вносит изменения, тестирует, коммитит: `git add . git commit -m "Add basic logging to Product and Order classes`
3. Отправляет ветку в удалённый репозиторий:  `git push -u origin feature/add-logging`
4. В GitHub создаётся Pull Request из `feature/add-logging` в `main`.
5. Назначаются ревьюеры, запускаются CI-пайплайны (проверка тестов, линтеров).
6. По результатам ревью вносятся правки (новые коммиты в ту же ветку): `git add . git commit -m "Fix logging format
   and add error handling" git push`
7. После одобрения PR, ветка сливается в `main` через кнопку в интерфейсе GitHub (merge commit или squash & merge).
8. Локально обновляется основная ветка: `git checkout main git pull`
9. Опционально — удаление локальной и удалённой ветки после мержа: `git branch -d feature/add-logging git push
   origin --delete feature/add-logging`
10. Перед началом новой задачи — убедиться, что `main` актуален, и создать новую feature-ветку от обновлённой основы.
